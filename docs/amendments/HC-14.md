# HC-14: Security Feature Flag Prohibition

**Status:** PROPOSED (pending CA ratification)
**Severity:** CONSTITUTIONAL
**Source:** hardened_mode=false incident
**Category:** Structural — prevents a class of bugs, not a specific bug

## Binding Text

Security properties SHALL NOT be gated by runtime configuration, feature
flags, environment variables, or boolean parameters.

All constitutional security invariants — including but not limited to
mlockall, PR_SET_DUMPABLE, PR_SET_NO_NEW_PRIVS, SSRF validation, handle
protocol enforcement, credential zeroization, forbidden header stripping,
and domain binding — MUST execute unconditionally in every non-test build.

Any code path that conditionally skips a security invariant based on a
runtime value is a constitutional violation requiring immediate halt
and CA review.

## Permitted Exceptions

- `#[cfg(test)]` blocks that bypass security for unit testing are
  permitted but MUST be documented with justification.
- Platform-conditional compilation (`#[cfg(target_os = "linux")]`) for
  OS-specific syscalls is permitted where the alternative is a compile
  error on unsupported platforms.

## Enforcement

1. **Static scan in CI:** grep-based detection of boolean flags, env var
   checks, and feature gates near security-critical functions. CI fails
   if patterns detected.
2. **Runtime verification:** scripts/verify-hardening.sh checks /proc
   state of live daemon. Runs in deployment pipeline.
3. **Code review:** Any PR adding a boolean parameter to a security
   function requires explicit constitutional justification.

## Rationale

On March 17, 2026, Wave 2 Phase C of the pre-deployment checklist
discovered that `hardened_mode: false` was hardcoded in
hermetic-daemon/src/lib.rs:119. This single boolean prevented all
three OS hardening measures (mlockall, PR_SET_DUMPABLE,
PR_SET_NO_NEW_PRIVS) from executing, despite:

- The hardening code being fully written and SM-1 ordered
- All unit tests passing (tests called functions directly, bypassing flag)
- Source inspection showing correct code
- Wave 1 codebase verification (80/80 PASS) not detecting the gap

The bug was only caught by live process inspection in Wave 2 Phase C.
This amendment prevents the entire class of "security code exists but
doesn't execute" bugs by prohibiting the pattern itself.

## Existing Code Debt

The following patterns predate this amendment and remain as structural
risk until refactored (requires separate CA authorization):

1. **`DaemonConfig.hardened_mode: bool`** (daemon.rs:83) — The field
   still exists. `#[derive(Default)]` defaults `bool` to `false`.
   Currently mitigated by production path (lib.rs:123) setting `true`.
2. **`daemon_run_inner(config, apply_hardening: bool)`** (daemon.rs:693)
   — Parameter allows callers to skip SM-1 hardening. Used with `false`
   only in `#[cfg(test)]` blocks (daemon.rs:1061, 1143).
3. **`start_daemon()` syslog gate** (daemon.rs:125) — Syslog check
   conditional on `hardened_mode`.

Recommended remediation: remove the `hardened_mode` field and
`apply_hardening` parameter entirely. Hardening should be called
unconditionally, with `#[cfg(test)]` used at the call site if tests
need to skip it.
